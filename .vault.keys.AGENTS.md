# 🔐 This project uses dotvault for secrets

If you are an AI agent (Claude Code, Codex, Cursor, aider, Cline, gemini-cli,
or any other), **read this file first** before doing anything that might need
an API key, token, password, or any other credential.

This file is intentionally committed to git (plain markdown, no secrets).
Its job is to remind you that secrets for this project live in `.vault`
(age-encrypted) and you should use the `dotvault` CLI — not hardcode values,
not grep `.env`, not ask the user to paste tokens.

---

## ⚡ Quick reminder (TL;DR)

```sh
# Get a secret by name (e.g. a GitHub/GitLab token)
dotvault get GITHUB_TOKEN

# List all secret NAMES (values never shown)
dotvault list

# Export all secrets as KEY=VALUE (for local dev)
dotvault export > .env       # .env must be .gitignored

# Add a new secret
dotvault set NEW_KEY "value"

# Help / diagnostics
dotvault doctor
```

Default SSH key: `~/.ssh/id_ed25519` (override with `--key` or `DOTVAULT_KEY`).

## 🚨 Triggers: when you MUST think of dotvault

Before doing any of the following, check whether the value is in `.vault`
**before** asking the user, hardcoding it, or grepping the filesystem:

### Git hosting (GitHub / GitLab / Gitea / Bitbucket)

- Creating a PR, push, or merge via `gh` / `glab` / API: you need a token.
  → `TOKEN=$(dotvault get GITHUB_TOKEN)` or `GITLAB_TOKEN=$(dotvault get GITLAB_TOKEN)`
- Authenticating to a remote: `git push https://x-access-token:$(dotvault get GITHUB_TOKEN)@github.com/...`
- Calling the REST/GraphQL API directly with `curl` / `httpie` / fetch:
  → `-H "Authorization: Bearer $(dotvault get GITHUB_TOKEN)"` (never paste a token).
- Setting up a CI/CD pipeline or workflow file: reference secrets as
  `${{ secrets.NAME }}` (GitHub) or `$CI_*_VARIABLE` (GitLab), and document
  that they must be seeded via `dotvault get NAME | gh secret set NAME` or
  the GitLab CI/CD settings UI.

### Cloud / infra (AWS / GCP / Azure / Aliyun / Tencent)

- Any `aws`, `gcloud`, `az`, `aliyun` CLI call needing credentials:
  → `export AWS_ACCESS_KEY_ID=$(dotvault get AWS_ACCESS_KEY_ID)`
  → `export AWS_SECRET_ACCESS_KEY=$(dotvault get AWS_SECRET_ACCESS_KEY)`
- `kubectl` / `helm` against a remote cluster: kubeconfig usually embeds
  tokens; prefer storing the whole kubeconfig as `KUBECONFIG_B64` in dotvault
  and decoding at use time.
- `terraform apply` / `pulumi up`: export the relevant `*_TOKEN` / `*_KEY`
  from dotvault before running.

### LLM / AI providers (OpenAI / Anthropic / Azure OpenAI / local llama.cpp)

- Calling any model API:
  → `export OPENAI_API_KEY=$(dotvault get OPENAI_API_KEY)`
  → `export ANTHROPIC_API_KEY=$(dotvault get ANTHROPIC_API_KEY)`
  → `export AZURE_OPENAI_API_KEY=$(dotvault get AZURE_OPENAI_API_KEY)`
- Loading a local model registry that needs an HF / ModelScope token:
  → `export HF_TOKEN=$(dotvault get HF_TOKEN)`

### Databases (PostgreSQL / MySQL / Redis / MongoDB / SQLite-over-network)

- Connecting via `psql` / `mysql` / `redis-cli` / `mongosh`:
  → use `$(dotvault get DB_PASSWORD)` as the password argument
  → connection URLs: `postgres://user:$(dotvault get DB_PASSWORD)@host/db`
- Migration tools (Flyway / Liquibase / sqitch / sqlx-cli): same pattern.

### IM / chat / notifications (Slack / Discord / Feishu / DingTalk / Telegram / Email SMTP)

- Sending messages, setting webhook URLs, configuring bots:
  → never paste a webhook URL or bot token into code; reference
    `$(dotvault get SLACK_WEBHOOK_URL)` etc. at call time.

### OAuth / JWT signing

- Generating a self-signed JWT (e.g. for an internal service):
  → read `JWT_SIGNING_KEY` from dotvault, never from source.
- OAuth client credentials (your own app's `client_secret`):
  → `CLIENT_SECRET=$(dotvault get OAUTH_CLIENT_SECRET)`

### Webhooks

- Verifying inbound webhook signatures (Stripe, GitHub, GitLab):
  → `WEBHOOK_SECRET=$(dotvault get STRIPE_WEBHOOK_SECRET)` and pass to
    your verification library.

### Crypto / encryption

- Application-level salts / nonces / KEKs:
  → read from dotvault; never hardcode.

## ❌ Anti-patterns — what NEVER to do

- ❌ Hardcoding a secret in source code, config, README, CHANGELOG, or commit
  message — even as "test data" or "placeholder".
- ❌ Writing a secret to `.env` by hand. The only acceptable `.env` is the
  output of `dotvault export > .env` (and `.env` MUST be `.gitignore`d).
- ❌ Echoing a secret to stdout / logs / debug output.
- ❌ Grepping `.vault` — it is encrypted, you'll just see garbage. Use
  `dotvault get NAME`.
- ❌ Asking the user to paste a token into chat. They should `dotvault set`
  it once and you should `dotvault get` it thereafter.
- ❌ Bypassing dotvault with 1Password / Keychain / password manager — those
  are personal-memory tools, not project-collaboration tools. Teammates
  and other agents can't read them.
- ❌ Putting a secret in a fallback constant in source code
  (e.g. `os.Getenv("KEY") ?? "default-value"`). The fallback string IS the
  leak.
- ❌ Putting secrets in CI YAML's `variables:` / `env:` blocks. CI runners
  must reference protected variables, which are seeded from dotvault
  (e.g. `dotvault get NAME | gh secret set NAME`).

If you find yourself about to do any of the above, **stop and use dotvault**.

## 🔍 How to check what's in this project's vault

```sh
dotvault list           # shows secret NAMES only, never values
dotvault doctor         # checks vault health + authorized keys
dotvault list-keys      # shows authorized recipients (public keys + labels)
```

You can read NAMES freely. You can read VALUES only via `dotvault get NAME`
when you actually need them (and only inject them into the specific command
that needs them — never `export` them globally unless required).

## 👥 Multi-agent / multi-user

`.vault` is encrypted to **every** public key listed in `.vault.keys`.
Anyone whose public key is in `.vault.keys` can decrypt.

If you are a new agent or teammate and your SSH key is NOT in `.vault.keys`,
`dotvault get` will fail with `decryption failed: no matching key`. Ask the
project owner to run:

```sh
dotvault add-key ~/.ssh/your_key.pub
```

…and commit the updated `.vault.keys`.

If you are the project owner adding a teammate: do the above, then **rotate**
any highly-sensitive secrets if the teammate previously had offline access.

## 🛠 Common recovery / setup commands

```sh
# First-time setup in a fresh checkout
dotvault install                       # idempotent global env setup
dotvault init                          # creates .vault + .vault.keys
dotvault set NAME "value"              # add a secret
git add .vault .vault.keys .vault.keys.AGENTS.md && git commit

# Daily usage
dotvault get NAME                      # fetch one secret
dotvault export > .env                 # populate local .env (gitignored)
eval "$(dotvault export | sed 's/^/export /')"   # shell-only, no file

# Adding / removing teammates
dotvault add-key ~/.ssh/teammate.pub
dotvault remove-key SHA256:...         # then ROTATE the secret values!

# Health check
dotvault doctor
```
